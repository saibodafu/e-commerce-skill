# 电商详情页内容黑白线框稿

## 触发条件

当用户希望梳理电商商品详情页内容时，使用本 skill。

只要用户的需求涉及以下任一方向，就应触发：

- 想为某个商品整理详情页内容结构
- 想把产品信息转成详情页文案初稿
- 想生成手机长图形式的详情页 HTML
- 想先看到一版低保真黑白线框稿效果
- 想围绕核心价值、用户痛点、产品卖点、使用场景、购买理由来组织详情页
- 想把零散商品信息整理成可交给运营、设计、内容团队继续执行的详情页策划稿

## 核心原则

- 先给结果，再进入共创。
- 第一阶段不追问，用户随便输入一点产品信息，也要直接生成初稿。
- 第二阶段按照固定步骤推进，每一层精修都先让用户补充更具体的信息，再给多个选择。
- 用户确认或选择后，再进入下一层。
- 所有精修内容最终要先整理成一个完整的详情页内容文档。
- 在内容文档确认后，再写成一份完整可运行的 HTML 进行呈现。

## 必要输入

第一阶段允许用户输入非常粗略的信息，不要求字段完整。常见输入包括：

```text
产品名：
产品定位：
目标人群：
核心场景：
核心卖点：
```

如果用户只给一句话、几个关键词、一个产品名或不完整卖点，也要继续生成初稿。

## 第一阶段：一键出初稿

### 执行方式

收到用户的初始产品信息后，直接生成一份完整 HTML，不先提出澄清问题。

如果信息不足，做轻量合理假设，并在 HTML 前用简短文字标注：

```text
以下初稿基于你提供的信息生成；缺失信息已做轻量假设，后续可以逐步精修。
```

不要虚构平台政策、认证、销量、转化率、行业 benchmark、未核验竞品数据或客户案例。

### HTML 视觉要求

生成的 HTML 必须可直接保存并在浏览器运行。页面风格遵循：

- 手机长图详情页形式，竖向滚动
- 主容器宽度控制在 `375px` 左右并居中展示
- 白底、黑字、浅灰模块
- 低保真黑白线框稿风格
- 每一屏之间用细黑线分隔
- 标题大、粗、短句化
- 用灰色卡片、圆形节点、简单线条图、流程图表达内容
- 不依赖外部图片、字体、CDN 或第三方库
- CSS 写在同一个 HTML 文件内

### 初稿页面结构

HTML 初稿必须包含以下模块：

1. 核心价值承诺层
2. 痛点锚定层
3. 卖点强化层
4. 使用场景层
5. 产品身份卡
6. 决策收口层

### 初稿生成逻辑

按以下内容逻辑组织初稿，但不要把长篇推理过程暴露给用户：

1. 从产品信息中提炼一句核心价值承诺，放在第一屏。
2. 围绕核心价值承诺提炼痛点、痒点、爽点，并转成详情页表达。
3. 将功能、成分、工艺、标准或差异化特征翻译成用户真正关心的购买理由。
4. 拆解产品可能出现的使用场景，筛选 Top 4 高价值场景。
5. 生成产品身份卡，让用户快速理解“这是什么、适合谁、解决什么、为什么买”。
6. 用决策收口层总结购买理由，降低用户下单前的犹豫。

### 初稿后的提问

生成 HTML 后，直接进入第二阶段的信息补充，不要问用户想精修哪一层。

使用以下固定提问：

```text
如果你想继续精修，请先补充更完整的产品信息。

产品信息如下：
【产品名】：
【产品定位】：
【目标人群】：
【核心场景】：
【核心卖点】：（请编号列出）
```

## 第二阶段：按顺序逐层精修

第二阶段必须按以下顺序推进，不让用户选择层级：

1. 核心价值承诺层
2. 痛点锚定层
3. 卖点强化层
4. 使用场景层
5. 产品身份卡
6. 决策收口层
7. 整理详情页内容大文档
8. 生成完整 HTML 黑白线框稿

每一层的固定节奏是：

1. 基于上一轮确认内容和用户补充信息进行分析。
2. 给多个可选方案。
3. 让用户选择、合并或提出修改方向。
4. 用户确认后，记录该层结果。
5. 自动进入下一层。

## 精修步骤

### 1. 核心价值承诺层

目标：将产品的所有卖点提炼成一句有差异点和共鸣感的核心价值承诺，适合放在电商详情页第一屏。

要求：

- 用词精练有力
- 朗朗上口
- 能激发目标人群共鸣
- 适合做第一屏关键定义

输出 10 句话备选，让用户选择或要求改写。

输出格式：

```text
核心价值承诺备选：
1.
2.
...
10.

请告诉我你选择哪一句，或说明你希望它更偏理性、情绪、专业、高端、性价比或生活方式。
```

### 2. 痛点锚定层

基于用户选中的核心价值承诺，挖掘用户的痛点、痒点、爽点。

分析维度：

- 痛点：用户在此情境中最担忧或最想解决的问题是什么？
- 痒点：用户希望呈现出的理想状态、身份感或角色投射是什么？
- 爽点：用户在哪一瞬间获得情绪满足、轻松感或被理解的感受？

然后提炼出适合电商详情页的一句话表达，给 10 个选择。

输出格式：

```text
痛点：

痒点：

爽点：

详情页表达备选：
1.
2.
...
10.

请告诉我你选择哪一句，或说明要更尖锐、更温和、更生活化还是更专业。
```

### 3. 卖点强化层

目标：精准翻译产品卖点，将功能转化成用户真正关心的购买理由。

对每一个卖点按以下步骤处理：

1. 提炼卖点：识别最重要的功能、成分、工艺、标准或差异化特征。
2. 转化买点视角：说明它能解决什么问题、带来什么好处。
3. 文案再造：写成一句更有感知力、更有画面感的详情页买点文案，每句控制在 25 字以内。

输出格式：

```text
卖点 1
主标题：用户买点
副标题：产品卖点

卖点 2
主标题：用户买点
副标题：产品卖点
```

如果用户给的卖点没有证据，不要补造数据。可以用体感、场景或结果表达，但避免虚假承诺。

### 4. 使用场景层

目标：把产品从抽象卖点落到真实购买和使用情境。

按以下步骤完成：

1. 基于产品信息和核心卖点，系统拆解可能出现的使用场景。
2. 根据市场大小、品牌优势和转化价值，筛选 Top 4 高价值场景。
3. 对 Top 4 高价值场景做极简概括。

输出格式：

```text
全量使用场景：
- 场景：
  价值：

Top 4 高价值场景：
1. 场景名：一句话概括
2. 场景名：一句话概括
3. 场景名：一句话概括
4. 场景名：一句话概括

请确认这 4 个场景是否保留，或告诉我想替换哪个。
```

### 5. 产品身份卡

目标：帮助用户快速判断这个产品是谁、解决什么、为什么值得买。

产品身份卡至少包含：

- 产品一句话定义
- 适合人群
- 核心场景
- 主要解决的问题
- 关键购买理由
- 不适合或需要谨慎承诺的边界

输出格式：

```text
产品身份卡：
产品定义：
适合人群：
核心场景：
解决问题：
购买理由：
边界提醒：

请确认这张身份卡是否准确，或告诉我哪一项需要改。
```

### 6. 决策收口层

目标：在详情页底部完成购买理由总结，降低用户犹豫。

可以从以下方向收口：

- 安心感
- 专业感
- 场景必要性
- 性价比
- 礼赠价值
- 低试错成本
- 长期使用价值

输出 3 到 5 组收口方案。

输出格式：

```text
决策收口备选：
1. 主标题：
   副标题：
   行动提示：

2. 主标题：
   副标题：
   行动提示：

请告诉我选择哪一组，或说明要更强转化、更克制、更高端还是更生活化。
```

## 最终交付：内容文档与 HTML

当六层内容都确认后，必须先将用户确认后的内容整理成一个完整的详情页内容大文档，再生成最终版 HTML。

内容大文档要清晰沉淀：

- 核心价值承诺
- 痛点、痒点、爽点
- 详情页表达句
- 每个卖点的产品卖点、用户买点、主标题、副标题
- Top 4 使用场景
- 产品身份卡
- 决策收口方案

内容大文档确认后，再输出 HTML。最终 HTML 要保留黑白线框稿形式，并明确展示：

- 首屏核心价值承诺
- 痛点、痒点、爽点或对应详情页表达
- 每个卖点的主标题和副标题
- Top 4 使用场景
- 产品身份卡
- 决策收口

输出 HTML 前，用一句话说明：

```text
下面先整理完整的详情页内容文档。确认后，我会把它写成可直接运行的 HTML 黑白线框稿。
```

当用户确认内容大文档后，再输出：

```text
下面是根据已确认内容生成的完整 HTML，可直接保存为 .html 文件运行。
```

## 默认处理方式

- 缺少非关键字段：直接做轻量假设，并标注“假设”。
- 缺少关键字段但仍能生成初稿：继续生成，不追问。
- 进入精修后信息不足：只围绕当前步骤提出必要补充，不跳到其他层。
- 用户中途要求重写某一层：只重写该层，并说明会在最终 HTML 中同步更新。
- 用户要求直接给最终版：可以跳过逐层确认，但要保留清晰假设。

## 质量标准

- 让用户第一眼看到一个完整、可运行、像详情页的 HTML 效果。
- 文案必须回到商品价值、人群需求、平台场景和转化路径。
- 不写泛泛的电商理论。
- 不堆空泛字段。
- 不编造未经核验的数据、政策、案例、认证或竞品事实。
- 页面结构要清楚，模块要像可交给设计师继续细化的低保真线框稿。
